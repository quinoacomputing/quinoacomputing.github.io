2 /*!
3  \file src/Control/Keywords.h
4  \copyright 2012-2015, J. Bakosi, 2016-2018, Los Alamos National Security, LLC.
5  \brief Definition of all keywords
6  \details This file contains the definition of all keywords, including those
7  of command-line argument parsers as well as input, i.e., control, file
8  parsers. All keywords are shared among all parsers of all executables.
9 
10  All keywords are case-sensitive.
11 
12  The information contained in this file is used to build data structures for
13  on-screen help on the command-line arguments and control file keywords,
14  available via the --help, --helpctr, and --helpkw command-line arguments.
15 
16  A note on design: Defining structs that have static member functions
17  returning a std::string is a way of storing C++-style strings at
18  compile-time (which is not possible in a straightforward manner at this
19  time). This could also be done with C-style const char* as well. The
20  '*_info' structs store these strings, which then is used to specialize the
21  _kw::keyword_ template, defined in Control/Keyword.h. Specializing the
22  _keyword_ template also requires a specification of the precise string of
23  characters that make up a keyword eventually matched by the parsers. Since
24  these are all template arguments, the construction of keywords, their help,
25  as well as all grammars, are entirely assembled at compile-time. Since the
26  '*_info' struct member functions are static, they can be called without
27  instantiating an object and thus available at compile-time.
28 
29  The definition of an '*_info' struct _requires_ at least the name, short,
30  and long descriptions, defined by member functions _name()_,
31  _shortDescription()_, and _longDescription()_, respectively. Everything else
32  is optional. However, when adding a new keyword it is highly recommended to
33  define all of the _optional_ members if they make sense for the given
34  keyword. If an expect value type is also given, that can be hooked up into
35  where it is used.
36 
37  The most general definition of a keyword is as follows:
38 
39  \code{.cpp}
40  // Keyword info definition
41  struct keyword_info {
42 
43  // Required very short name, usually a single word or (for e.g.
44  // policies) a single character. This can be the keyword itself, but
45  // does not have to be. This field is used as an id of the option or
46  // setting.
47  static std::string name() { return "Name"; }
48 
49  // Required short keyword description
50  static std::string shortDescription() { return "Short description"; }
51 
52  // Required detailed keyword description. This returns a string literal,
53  // since this is usually multi-line and it is less work to maintain this
54  // way.
55  static std::string longDescription() { return
56  R"(Longer, possibly multi-line description of the keyword. Example
57  usage of the keyword is welcome here. Don't worry about formatting:
58  when this field is printed, extra spaces will be removed and line
59  breaks will be inserted.)";
60  }
61 
62  // Optional keyword alias. See also kw::Alias and
63  // <tpl_install_dir>/include/pegtl/pegtl/constants.hh for examples
64  // of what can be passed as template arguments (basically single
65  // characters). The one below defines the character 'c' as the alias.
66  // Aliases are single character long. The idea of an alias is to have a
67  // long as well as a short keyword for the same functionality.
68  // Currently, this is only hooked up for command-line arguments and not
69  // for control-file keywords, which is intentional. Command-line
70  // arguments are a lot less than control file keywords and are more
71  // frequently typed by the user. Thus command-line argument aliases are
72  // user-friendly. There are many control file keywords and aliases would
73  // only cause confusion. Defining an alias for a command-line argument
74  // enables the command-line parser to match on '--longer_keyword' as
75  // well as on '-c'. Depending on whether the alias typedef is defined
76  // for a keyword or not, the correct grammar is automatically generated
77  // at compile-time, matching on both the longer keyword as well as on
78  // the alias. Defining an alias for a control file keyword can be done
79  // but has no effect in a control file parser.
80  using alias = Alias< c >;
81 
82  // Optional single-character (policy) code. See also kw::Code and
83  // <tpl_install_dir/include/pegtl/pegtl/constants.hh for examples
84  // of what can be passed as template arguments (basically single
85  // characters). The one below defines the character 'C' as the (policy)
86  // code. This code is used for abbreviating policies used to configure
87  // various orthogonal behaviors of classes using policy-based design.
88  using code = Code< C >;
89 
90  // Optional expected data for the keyword - bundled to struct expect.
91  // This struct is entirely optional within a keyword definition.
92  // However, if it is defined, it must at least define the static member
93  // function description() which returns the description of the type the
94  // keyword expects. It may also optionally define the following fields:
95  //
96  // - type - defining the expected type
97  // - lower - lower bound of the expected value
98  // - upper - upper bound of the expected value
99  // - choices - valid choices for the expected value
100  //
101  struct expect {
102 
103  // If this struct is defined, required expected type description, max
104  // 10 characters long
105  static std::string description() { return "int"; }
106 
107  // Optional expected type
108  using type = std::streamsize;
109 
110  // Optional expected value lower bound
111  static const type lower = 1;
112 
113  // Optional expected value upper bound
114  static const type upper = std::numeric_limits< tk::real >::digits10 + 1;
115 
116  // Optional expected valid choices description, here giving
117  // information on the expected type and the valid bounds. Note that
118  // this can be any string, but if they exist, it is a good idea give
119  // at least some information on the bounds, as below, since the bounds
120  // are NOT displayed in the help for a keyword. This decision keeps
121  // the bounds specifications generic since they can be any type. As a
122  // result, the help structures, defined in HelpFactory.h, are simpler
123  // as they do not have to be parameterized by the type of the bounds,
124  // which greatly simplifies that code.
125  static std::string choices() {
126  return "integer [" + std::to_string(lower) + "..." +
127  std::to_string(upper) + ']';
128  }
129 
130  };
131 
132  };
133 
134  // Keyword definition, passing the above info struct as the first
135  // template argument, and the rest of the template arguments are the
136  // characters of the keyword to be matched by the parser. Remember: all
137  // keywords are case-sensitive. This one contrived example below defines
138  // keyword 'kw', matching the keyword 'KeYwOrD'.
139  using kw = keyword< keyword_info, K,e,Y,w,O,r,D >;
140  \endcode
141  \see Control/Keyword.h
142  \see Control/HelpFactory.h
143 */
